# Death-Claim Deferred Hardening Items

> **Purpose:** Track the known unknowns that are intentionally deferred while the death-claim scenario is translated into downstream structural artifacts.

---

**Status:** Active PoC planning artifact

This file exists to make the current deferments explicit. These are not accidental omissions. They are bounded hardening items with clear triggers.

| Defer item                                             | Reason                                                                                                                                                                                                                                 | Hardening trigger                                                                                                                                        |
|--------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------|
| Exact missing-vs-ambiguous boundary                    | The workshop spec proves the three-case pattern, but the precise disposition boundary still needs claims-ops and governance alignment before it becomes architecture-driving logic.                                                    | Before the scenario is translated into concrete routing or escalation rules in the vertical slice or downstream Tree A example.                          |
| Exact tone markers for claimant-facing follow-up       | The workshop spec establishes empathy and operational appropriateness, but not the final review rubric for claimant-facing language.                                                                                                   | Before follow-up-message prompts, templates, or review criteria are treated as stable demo behavior.                                                     |
| Exact governance/data-science review metrics           | The workshop spec requires inspectability, but does not yet lock the scorecard or demo review signals.                                                                                                                                 | Before eval, observability, or demo instrumentation are treated as acceptance-driving outputs.                                                           |
| Exact confidence/reviewability model                   | The scenario now needs an explicit reviewability signal, but the final rubric, weighting, and thresholds are not yet hardened.                                                                                                         | Before confidence/reviewability is treated as a stable acceptance or demo-governing output.                                                              |
| PII in graph state and `PIIGuardrailAdapter` placement | For PoC speed and to secure early wins using LangGraph's accumulated state abstractions, raw PII remains in the graph state initially. It is stripped explicitly via `PIIGuardrailAdapter` immediately prior to external model action. | Before we deploy a robust local SLM to pre-sanitize all incoming document intake contexts upstream of the LangGraph execution.                           |
| Strict runtime state validation (e.g., Pydantic)       | We are using `TypedDict` for the graph state for ergonomics and speed. While native to LangGraph, it lacks deep runtime type-checking during node transitions.                                                                         | Before transitioning the PoC to a production-grade pipeline where strict schema guarantees between nodes are mandated.                                   |
| Granular graph node tracing (1:1 Use Case Mapping)     | For PoC demo simplicity and structural legibility, we chose "Option A: Aggregate Nodes by Phase." This hides the strict 1:1 Use Case mapping inside aggregate phases.                                                                  | As an immediate next step post steel-thread functionality, before full observability/tracing of individual internal logic paths is required.             |
| LLM Fact Extraction + LLM Generation                   | "Option A: LLM Artifact Generation Only" is used now to guarantee unconditionally green tests. "Option B: LLM Fact Extraction" is deferred due to hallucination and PII leakage risks during extraction.                               | After we have correctly leveraged the local SLM and ensured no raw PII is going to the external model that would be non-compliant.                       |
| Fake System Collaborator Adapters                      | The `PIIGuardrailAdapter`, `PolicyLookup`, `DocumentIntake`, and `ReviewQueue` adapters are fake-backed to stabilize the flow without needing the physical enterprise footprint.                                                       | Before integrating with live enterprise orchestration endpoints or databases for Policy Admin or Case Management.                                        |
| LangGraph Thread Checkpointing / Persistence           | Broad persistence or data stores are explicitly out of scope for block 4/5. LangGraph thread state will run ephemerally in-memory.                                                                                                     | Before requiring durable pause-and-resume workflows or human-in-the-loop (HITL) handoffs that survive server restarts.                                   |
| Synthetic Data Pipeline Generation                     | Canonical hand-authored fixtures are sufficient for the steel-thread. A synthetic data pipeline is deferred to avoid building unrelated infrastructure first.                                                                          | Before scaling evaluation, testing, or model fine-tuning beyond the 3 canonical representative cases.                                                    |
| Secrets Management for Model API Keys                  | Explicit `.env` mapping (`LLM_MAIN_API_KEY`) is used for the PoC to unblock rapid local iteration and explicit routing.                                                                                                                | Before deploying to any shared environment, these keys must be migrated to a secure provider (ExternalSecrets, 1Password, Azure Key Vault).              |
| Distributed Edge Rate Limiting                         | Local or inline rate-limiting (currently in `LiveChatModelAdapter` via `InMemoryRateLimiter`) is sufficient to protect against PoC demo limits. Full distributed queueing or distributed rate limiting (e.g. Redis) is deferred.       | Before exposing the triage pathway to production workloads where concurrent node executions exceed standard provider limits.                             |
| Universal Framework Guardrails                         | The PoC relies on prompt engineering and the Vaultless DSPy PII Guardrail for core safety. Adding NeMo Guardrails or LlamaGuard directly to the `LiveChatModelAdapter` adds unacceptable latency and deployment surface for the demo.  | Before the chatbot is exposed to external customers directly where prompt injection and arbitrary toxicity represent immediate enterprise risk.          |
| Multimodal PII Extraction (Llama 3.2 Vision)           | Llama 3.1 8B is sufficient for text-based PII extraction and fits within local machine memory/performance constraints. Moving to Llama 3.2 11B for vision capabilities introduces significant performance and size overhead.           | Before the scenario is expanded to require PII extraction directly from scanned documents or images (like photos of IDs) instead of pre-parsed OCR text. |
| DSPy Guardrail Training Data Size                      | A microscopic dummy dataset is used to prove the compilation pipeline works.                                                                                                                                                           | Before deploying the compiled PII guardrail, it must be retrained on a statistically significant dataset that reflects real-world industry PII patterns. |
