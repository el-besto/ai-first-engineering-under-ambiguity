# DSPy Guardrail: Phase 3 Implementation Plan (Live E2E Testing)

## Phase Summary

- **Feature:** Local SLM PII Guardrail
- **Phase Goal:** Establish a dedicated E2E test that validates the full LangGraph routing workflow using the *live* local SLM (`VaultlessPIIGuardrail`) in combination with the *live* OpenAI API, proving the encryption/decryption round-trip works against real LLM outputs.
- **Status:** ready (stretch goal)

## Contract Inputs

This phase must adhere to the following established boundaries:

- **Workshop spec:** `plan/death-claim/workshop-spec.md`
- **Technical Architecture:** `plan/technical/pii-guardrail-architecture.md`
- **Reference Code:** `plan/technical/pii-guardrail-code-reference.md`

## Out Of Scope

Explicitly what this phase will *not* accomplish:

- Training or compiling the DSPy model (completed in DSPy Phase 1).
- Wiring the guardrail into the main application dependencies for Streamlit/FastAPI (completed in DSPy Phase 2).
- Mocked testing (this phase is strictly for live model execution).

## Target Production Surface

The minimum supporting surface required to satisfy this phase:

### New Files

- `tests/acceptance/features/test_live_dspy_e2e_triage.py` - The core E2E suite that executes the `TriageGraph` with both the local `ollama` DSPy extractor and the `OpenAI` generator active.

### Modified Files

- `Makefile` - Update or add commands to ensure the `ollama` service is running before executing this specific test suite.

## Assertions & Behavior Contracts

- The tests MUST load the live `.env` variables for both the Guardrail (AES Key, SLM endpoint) and the Generator (OpenAI Key).
- The `test_live_dspy_e2e_triage.py` suite MUST be decorated with `@pytest.mark.live` and potentially a new `@pytest.mark.dspy` marker to allow granular execution.
- The tests MUST explicitly assert the Privacy Boundary during the run:
  - `TriageGraphState.claim_bundle` (post-guardrail) MUST NOT contain raw PII strings (e.g., "John Doe").
  - `TriageGraphState.claim_bundle` MUST contain `TOK-...` base64 strings.
- The tests MUST assert the final `TriageResult` generated by OpenAI has successfully reversed the `TOK-...` strings back to original cleartext PII (e.g., "John Doe" appears in the `case_summary`).

## Ordered Implementation Steps

Implement in this order:

1. Update `pytest.ini` to register a `dspy` test marker (if desired for granularity).
2. Author `tests/acceptance/features/test_live_dspy_e2e_triage.py`.
3. Configure the test fixture to override dependencies with BOTH `LiveChatModelAdapter` and `VaultlessPIIGuardrail`.
4. Implement the assertion logic to prove raw PII is stripped pre-generation and restored post-generation.
5. Run the test against a locally running `llama3.1:8b` container and the live OpenAI API.

## Verification Plan

Verify the completion of this phase with evidence that:

- Review `docs/patterns.md` against the implemented changes. If any rule or architectural pattern is violated, halt and prompt the user to decide on resolution versus explicit waiver.
- The test executes successfully, passing the mid-flight `state` inspections for PII redaction and the final cleartext assertions.

- The developer executing this test has the `ollama` container running locally on port `11434` with the compiled DSPy JSON trace available.
- The developer has a valid OpenAI API key.
